# Requirements Document

## Introduction

WeDo is a couples-only mobile app (React Native / Expo) that helps partners track, celebrate, and deepen their relationship through shared digital experiences. The app spans ten major feature areas: (1) an Adventure Log timeline feed where couples preserve completed dates as interactive memory entries with scratch-off reveals and optional premium audio notes; (2) an onboarding and pairing flow that links two users via a 6-digit code into a permanent relationship; (3) a Privacy Vault with biometric lock that secures the app when backgrounded; (4) an Interactive Sticker Calendar where partners drag-and-drop stickers onto days with real-time sync; (5) a Shared Bucket List with a Social Catcher that captures URLs from the device Share Sheet; (6) an Indecision Wheel that randomly picks a bucket-list item and notifies the partner; (7) Home Screen Widgets displaying a "days together" counter on iOS (SwiftUI) and Android; (8) a Deep Connection Conversation Deck of 300+ swipeable prompt cards (premium); (9) premium entitlement logic via RevenueCat for a $4.99 lifetime unlock; and (10) a five-tab navigation architecture built on react-navigation. The backend is Supabase (Auth, Database, Storage, Realtime) and state management uses Zustand.

## Glossary

- **Adventure_Log**: The scrollable vertical feed displaying all completed relationship memories in reverse-chronological order, rendered in the Timeline tab (Tab 1).
- **Memory_Entry**: A single record in the Adventure Log consisting of a compressed photo, caption text, creation timestamp, scratch-off state, and an optional audio note.
- **Scratch_Off_Overlay**: A digital silver foil layer covering a Memory_Entry photo that the viewing partner removes by rubbing the screen via touch gesture.
- **Image_Compressor**: The on-device module (expo-image-manipulator) responsible for resizing and compressing photos before upload.
- **Audio_Recorder**: The on-device module (expo-av) responsible for recording, encoding, and uploading voice notes in .m4a format.
- **Supabase_Auth**: The Supabase authentication service supporting Apple, Google, and Magic Link sign-in methods.
- **Supabase_Storage**: The remote file storage backend for compressed photos, audio files, and custom sticker images.
- **Supabase_DB**: The remote PostgreSQL database storing all app data including Memory_Entry metadata, pairing records, bucket list items, calendar stickers, and real-time sync state.
- **Partner**: The other user linked via the shared Relationship_ID.
- **Relationship_ID**: The unique identifier linking two paired users, created during the pairing flow.
- **Pairing_Code**: A 6-digit alphanumeric code generated by User 1 during onboarding that User 2 inputs to establish the Relationship_ID.
- **Pairing_Gateway**: The onboarding screen presented after authentication where users choose "Start New Adventure" (generate code) or "Join Partner" (input code).
- **Privacy_Vault**: The biometric lock screen (FaceID/TouchID/PIN) that secures the app when it returns from the background, using expo-local-authentication.
- **Blur_Overlay**: A full-screen blurred layer (expo-blur) displayed over app content while the Privacy_Vault authentication prompt is active.
- **Sticker_Calendar**: The full-screen monthly calendar view (Tab 2) where partners drag-and-drop stickers onto specific days.
- **Sticker_Drawer**: The bottom sheet UI containing available stickers that the user drags onto the Sticker_Calendar.
- **Default_Sticker**: One of the 20 pre-loaded sticker images available to all users at no cost.
- **Custom_Sticker**: A user-uploaded cropped photo used as a draggable calendar sticker, available to Premium_User only.
- **Bucket_List**: The shared, real-time synced to-do list (Tab 3) where couples track planned dates and activities, capped at 10 items for Free_User.
- **Social_Catcher**: The native intent filter (Expo Config Plugin) that intercepts URLs shared from external apps (e.g., Instagram, Threads) and routes them into the Bucket_List via the AddToListModal.
- **AddToListModal**: A full-screen slide-up modal for adding a new item to the Bucket_List, either manually or via the Social_Catcher.
- **Indecision_Wheel**: A physics-based roulette wheel that randomly selects one uncompleted Bucket_List item, accessed from the Bucket List tab header.
- **Home_Screen_Widget**: A native iOS (SwiftUI) or Android widget displaying a running "days together" counter, reading from local storage.
- **Conversation_Deck**: A premium-only swipeable card deck (Tab 4) containing 300+ intimate conversation starter prompts loaded from a local JSON file.
- **Paywall_Modal**: A full-screen modal presented when a Free_User attempts a premium-only action, offering the $4.99 lifetime unlock via RevenueCat.
- **RevenueCat**: The third-party SDK managing in-app purchase entitlements and subscription restoration.
- **Premium_User**: A user who has purchased the $4.99 lifetime unlock, verified via RevenueCat entitlement `premium_lifetime`.
- **Free_User**: A user who has not purchased the premium unlock.
- **Root_Navigator**: The top-level react-navigation Native Stack containing OnboardingStack, MainTabNavigator, PaywallModal, and AddToListModal.
- **MainTabNavigator**: The bottom tab navigator with five tabs: Timeline, Calendar, Bucket List, Connection, and Settings.
- **Settings_Screen**: Tab 5 providing user profile, partner info, app preferences, subscription management, Privacy_Vault toggle, theme selection, and widget installation instructions.
- **Zustand_Store**: The client-side state management store (Zustand) holding auth state, premium entitlement status, and ephemeral UI state.

## Requirements

### Requirement 1: Display the Adventure Log Feed

**User Story:** As a couple, we want to see all our completed date memories in a scrollable timeline, so that we can revisit our shared experiences together.

#### Acceptance Criteria

1. WHEN the user navigates to the Timeline tab, THE Adventure_Log SHALL render all Memory_Entry records for the active Relationship_ID in reverse-chronological order.
2. WHILE the Adventure_Log contains more entries than fit on screen, THE Adventure_Log SHALL support vertical scrolling with smooth momentum.
3. THE Adventure_Log SHALL display each Memory_Entry with the compressed photo, caption text, and formatted creation timestamp.
4. WHEN no Memory_Entry records exist for the Relationship_ID, THE Adventure_Log SHALL display an empty-state illustration with the message "No adventures yet — complete a date from your Bucket List to start!"
5. WHEN a new Memory_Entry is created by either partner, THE Adventure_Log SHALL appear on both partners' feeds via Supabase real-time sync within 3 seconds on a stable connection.

### Requirement 2: Create a New Memory Entry

**User Story:** As a user, I want to create a memory from a completed Bucket List item, so that I can preserve the experience in our shared timeline.

#### Acceptance Criteria

1. WHEN a Bucket List item is marked as completed, THE Adventure_Log SHALL present a memory creation modal prompting for a photo and caption.
2. THE Memory_Entry creation modal SHALL require exactly one photo selection from the device gallery or camera.
3. THE Memory_Entry creation modal SHALL require a caption between 1 and 500 characters.
4. WHEN the user submits a valid photo and caption, THE Adventure_Log SHALL create a new Memory_Entry linked to the active Relationship_ID with a server-generated timestamp.
5. IF the user cancels the memory creation modal, THEN THE Adventure_Log SHALL discard unsaved input and return to the previous screen without creating a Memory_Entry.
6. IF the photo upload fails due to a network error, THEN THE Adventure_Log SHALL display an inline error message "Upload failed — please check your connection and try again" and retain the user's input for retry.

### Requirement 3: Compress Photos Before Upload

**User Story:** As a couple, we want our photos automatically optimized, so that storage costs stay low and the app remains fast.

#### Acceptance Criteria

1. WHEN a photo is selected for a Memory_Entry, THE Image_Compressor SHALL resize the photo so the longest dimension does not exceed 1200 pixels while preserving the original aspect ratio.
2. WHEN a photo is selected for a Memory_Entry, THE Image_Compressor SHALL compress the photo to JPEG format with a quality factor of 0.7.
3. THE Image_Compressor SHALL produce an output file no larger than 200KB.
4. IF the Image_Compressor cannot reduce the file to 200KB at quality 0.7, THEN THE Image_Compressor SHALL iteratively lower the quality factor in steps of 0.1 down to a minimum of 0.3 until the file is within 200KB.
5. IF the Image_Compressor cannot reduce the file to 200KB at quality 0.3, THEN THE Image_Compressor SHALL reject the photo and display the message "This photo is too large to process — please choose another."
6. FOR ALL valid photos, compressing then uploading then downloading then displaying SHALL produce a visually equivalent image to the compressed original (round-trip integrity).

### Requirement 4: Scratch-Off Reveal Mechanic

**User Story:** As a user, I want to cover my memory in a digital scratch-off foil so my partner has a fun, tactile way to reveal it.

#### Acceptance Criteria

1. WHEN a new Memory_Entry is created, THE Scratch_Off_Overlay SHALL cover the Memory_Entry photo with a silver foil texture by default (revealed state: false).
2. WHEN the Partner touches and drags across the Scratch_Off_Overlay, THE Scratch_Off_Overlay SHALL progressively erase the foil along the touch path, revealing the photo beneath.
3. WHEN 60% or more of the Scratch_Off_Overlay area has been erased by the Partner, THE Scratch_Off_Overlay SHALL auto-complete the reveal with a fade-out animation and set the Memory_Entry revealed state to true.
4. WHEN the revealed state changes to true, THE Supabase_DB SHALL persist the updated state and sync it to the creating partner's device via real-time subscription.
5. THE Scratch_Off_Overlay SHALL only be interactive for the Partner who did not create the Memory_Entry.
6. WHEN the creating user views their own Memory_Entry, THE Adventure_Log SHALL display the photo without a Scratch_Off_Overlay.
7. WHILE the Scratch_Off_Overlay is being erased, THE Adventure_Log SHALL trigger subtle haptic feedback on each touch-move event using expo-haptics.

### Requirement 5: Audio Memory Attachment (Premium)

**User Story:** As a premium user, I want to attach a voice note to a memory, so that I can add a personal, emotional layer to our shared moments.

#### Acceptance Criteria

1. WHERE the user is a Premium_User, THE Adventure_Log SHALL display a microphone icon on each revealed Memory_Entry allowing audio recording.
2. WHEN a Free_User taps the microphone icon, THE Adventure_Log SHALL display the premium paywall modal.
3. WHEN a Premium_User taps the microphone icon, THE Audio_Recorder SHALL begin recording audio in .m4a format (MPEG4 AAC encoding).
4. WHILE the Audio_Recorder is recording, THE Audio_Recorder SHALL display a live duration counter and a stop button.
5. WHEN the recording duration reaches 60 seconds, THE Audio_Recorder SHALL automatically stop recording.
6. WHEN recording stops, THE Audio_Recorder SHALL upload the .m4a file to Supabase_Storage and link it to the Memory_Entry.
7. IF the audio upload fails, THEN THE Audio_Recorder SHALL retain the local recording file and display "Upload failed — tap to retry."
8. WHEN a Memory_Entry has an attached audio note, THE Adventure_Log SHALL render a waveform visualization and a play/pause button on that entry.
9. THE Adventure_Log SHALL allow only one audio note per Memory_Entry.

### Requirement 6: Real-Time Sync Between Partners

**User Story:** As a couple, we want our Adventure Log to stay in sync in real time, so that both of us always see the latest memories and scratch-off states.

#### Acceptance Criteria

1. WHEN the Adventure_Log is open, THE Supabase_DB SHALL maintain an active real-time WebSocket subscription scoped to the active Relationship_ID.
2. WHEN a Memory_Entry is created by one partner, THE Adventure_Log on the other partner's device SHALL display the new entry with a spring entrance animation.
3. WHEN a Scratch_Off_Overlay revealed state changes to true, THE Adventure_Log on the creating partner's device SHALL update the entry to show the revealed photo.
4. WHEN a new audio note is attached to a Memory_Entry, THE Adventure_Log on the Partner's device SHALL display the waveform and play button without requiring a manual refresh.
5. IF the WebSocket connection is lost, THEN THE Adventure_Log SHALL display a subtle "Reconnecting..." indicator and automatically re-establish the subscription when connectivity returns.
6. WHEN the WebSocket connection is re-established, THE Adventure_Log SHALL reconcile any missed updates from Supabase_DB.

### Requirement 7: Memory Entry Data Persistence

**User Story:** As a user, I want my memories safely stored and retrievable, so that our Adventure Log is never lost.

#### Acceptance Criteria

1. THE Supabase_DB SHALL store each Memory_Entry with the following fields: id (UUID), relationship_id, created_by (user_id), photo_url, caption, created_at (timestamp), revealed (boolean), and audio_url (nullable).
2. THE Adventure_Log SHALL enforce row-level security so that only users belonging to the Memory_Entry's Relationship_ID can read or write that entry.
3. WHEN a Memory_Entry photo is uploaded, THE Supabase_Storage SHALL store the file in a path scoped to the Relationship_ID (e.g., `{relationship_id}/memories/{entry_id}.jpg`).
4. WHEN a Memory_Entry audio note is uploaded, THE Supabase_Storage SHALL store the file at `{relationship_id}/audio/{entry_id}.m4a`.
5. FOR ALL Memory_Entry records written to Supabase_DB, reading the record back SHALL return field values identical to those written (round-trip data integrity).

### Requirement 8: Adventure Log Visual Design

**User Story:** As a user, I want the Adventure Log to feel warm and intimate, so that revisiting memories is an emotionally rich experience.

#### Acceptance Criteria

1. THE Adventure_Log SHALL render on a Charcoal (#121212) background consistent with the app-wide dark theme.
2. THE Adventure_Log SHALL display Memory_Entry captions in a clean Sans-Serif typeface and Memory_Entry titles or date headers in an elegant Serif typeface.
3. THE Adventure_Log SHALL apply glassmorphism styling (semi-transparent background with blur) to Memory_Entry cards.
4. WHEN a new Memory_Entry appears via real-time sync, THE Adventure_Log SHALL animate the entry in using a spring animation (react-native-reanimated).
5. THE Scratch_Off_Overlay foil texture SHALL use a silver metallic gradient consistent with a physical scratch card appearance.
6. THE Adventure_Log SHALL use Soft Coral (#FF7F50) for interactive action elements (e.g., create button, microphone icon) and Teal (#40E0D0) for milestone indicators (e.g., date headers, timestamps).

### Requirement 9: Onboarding and Pairing

**User Story:** As a new user, I want to sign up and pair with my partner using a simple code, so that we can start sharing our relationship experiences immediately.

#### Acceptance Criteria

1. WHEN the app is launched for the first time, THE Root_Navigator SHALL display a Splash Screen followed by the Auth Screen.
2. THE Supabase_Auth SHALL support sign-up and sign-in via Apple OAuth, Google OAuth, and Magic Link (email).
3. WHEN the user completes authentication, THE Pairing_Gateway SHALL present two options: "Start New Adventure" and "Join Partner."
4. WHEN the user selects "Start New Adventure," THE Pairing_Gateway SHALL generate a unique 6-digit alphanumeric Pairing_Code and display it on screen for the user to share with the Partner.
5. WHEN the user selects "Join Partner," THE Pairing_Gateway SHALL present a text input field for entering a 6-digit Pairing_Code.
6. WHEN User 2 submits a valid Pairing_Code, THE Supabase_DB SHALL create a new Relationship_ID record linking both user_ids permanently.
7. IF User 2 submits an invalid or expired Pairing_Code, THEN THE Pairing_Gateway SHALL display the message "Invalid code — please ask your partner for a new one."
8. WHEN pairing succeeds, THE Pairing_Gateway SHALL play a success animation (confetti particles and device haptic feedback) and route both users to the MainTabNavigator with the Timeline tab active.
9. WHILE a user is authenticated but not yet paired, THE Root_Navigator SHALL prevent navigation to the MainTabNavigator and keep the user on the Pairing_Gateway.

### Requirement 10: Privacy Vault and Biometric Lock

**User Story:** As a user, I want the app to lock automatically when I leave it, so that our private memories stay secure if someone else picks up my phone.

#### Acceptance Criteria

1. WHEN the app transitions from the active state to the background state (React Native AppState API), THE Privacy_Vault SHALL activate and cover the entire screen with the Blur_Overlay.
2. WHEN the user returns to the app while the Privacy_Vault is active, THE Privacy_Vault SHALL prompt for biometric authentication using expo-local-authentication authenticateAsync().
3. WHEN biometric authentication succeeds, THE Privacy_Vault SHALL dismiss the Blur_Overlay and reveal the last viewed tab.
4. IF biometric authentication fails, THEN THE Privacy_Vault SHALL offer a device PIN fallback via the fallbackLabel option.
5. IF the device does not support biometric authentication, THEN THE Privacy_Vault SHALL fall back to device passcode authentication.
6. THE Settings_Screen SHALL provide a toggle to enable or disable the Privacy_Vault feature.
7. WHILE the Privacy_Vault toggle is disabled in Settings, THE Privacy_Vault SHALL not activate when the app transitions to the background.
8. WHILE the Blur_Overlay is visible, THE Privacy_Vault SHALL prevent interaction with any underlying app content.

### Requirement 11: Interactive Sticker Calendar

**User Story:** As a couple, we want a shared calendar where we can place fun stickers on special days, so that we can visually mark and celebrate our moments together.

#### Acceptance Criteria

1. WHEN the user navigates to Tab 2 (Calendar), THE Sticker_Calendar SHALL render a full-screen monthly calendar view using react-native-calendars.
2. THE Sticker_Calendar SHALL allow horizontal swiping to navigate between months.
3. THE Sticker_Drawer SHALL be accessible by dragging up from the bottom of the Sticker_Calendar screen.
4. THE Sticker_Drawer SHALL contain 20 Default_Sticker images available to all users.
5. WHEN the user drags a sticker from the Sticker_Drawer onto a calendar day cell, THE Sticker_Calendar SHALL capture the exact X and Y drop coordinates within that cell using react-native-gesture-handler.
6. WHEN a sticker is dropped on a day cell, THE Supabase_DB SHALL persist the sticker record (sticker_id, day, x_coordinate, y_coordinate, relationship_id, placed_by) in real time.
7. WHEN a sticker record is persisted, THE Sticker_Calendar on the Partner's device SHALL display the sticker at the saved coordinates with a spring entrance animation.
8. THE Sticker_Calendar SHALL allow multiple stickers to be placed on the same day cell.
9. IF the sticker drop fails due to a network error, THEN THE Sticker_Calendar SHALL display an inline error message and revert the sticker to the Sticker_Drawer.
10. FOR ALL sticker placements written to Supabase_DB, reading the record back SHALL return coordinates and metadata identical to those written (round-trip data integrity).

### Requirement 12: Shared Bucket List and Social Catcher

**User Story:** As a couple, we want a shared to-do list for date ideas that can also capture links we share from social media, so that we never lose inspiration for our next adventure.

#### Acceptance Criteria

1. WHEN the user navigates to Tab 3 (Bucket List), THE Bucket_List SHALL display all items for the active Relationship_ID in the order they were added.
2. THE Bucket_List SHALL sync items between both partners in real time via Supabase real-time subscription.
3. WHEN the user taps the floating action button ("+"), THE AddToListModal SHALL open for manual item entry.
4. WHILE the user is a Free_User, THE Bucket_List SHALL enforce a maximum of 10 items.
5. IF a Free_User attempts to add an 11th item, THEN THE Bucket_List SHALL display the Paywall_Modal instead of the AddToListModal.
6. WHEN the user shares an Instagram or Threads URL from the device Share Sheet, THE Social_Catcher SHALL intercept the intent and open the AddToListModal with the URL pre-populated.
7. THE Social_Catcher SHALL use an Expo Config Plugin to register native intent filters for URL sharing on both iOS and Android.
8. WHEN the user submits the AddToListModal, THE Supabase_DB SHALL persist the new Bucket_List item (id, relationship_id, title, url (nullable), completed (boolean), created_at, created_by).
9. WHEN either partner marks a Bucket_List item as completed, THE Bucket_List SHALL update the item's completed state on both devices via real-time sync.
10. IF the Bucket_List item save fails due to a network error, THEN THE Bucket_List SHALL display an inline error message "Save failed — please check your connection and try again" and retain the user's input.

### Requirement 13: Indecision Wheel

**User Story:** As a couple, we want a fun way to randomly pick a date idea when we can't decide, so that we spend less time debating and more time adventuring.

#### Acceptance Criteria

1. WHEN the user taps the "Indecision Wheel" icon in the top-right of the Bucket List tab header, THE Root_Navigator SHALL push the WheelScreen onto the navigation stack.
2. THE Indecision_Wheel SHALL render a physics-based roulette wheel populated with all uncompleted Bucket_List items for the active Relationship_ID.
3. WHEN the user spins the Indecision_Wheel, THE Indecision_Wheel SHALL animate a deceleration curve and land on one randomly selected uncompleted Bucket_List item.
4. WHEN the Indecision_Wheel lands on an item, THE Indecision_Wheel SHALL trigger strong haptic feedback using expo-haptics (NotificationFeedbackType.Success).
5. WHEN the Indecision_Wheel lands on an item, THE Indecision_Wheel SHALL send a push notification to the Partner with the selected item's title.
6. IF the Bucket_List contains zero uncompleted items, THEN THE Indecision_Wheel SHALL display the message "Add some ideas to your Bucket List first!" and disable the spin action.
7. THE Indecision_Wheel SHALL display the selected item prominently after the spin completes, with an option to mark it as completed.

### Requirement 14: Home Screen Widgets

**User Story:** As a couple, we want a home screen widget showing how many days we've been together, so that we get a daily reminder of our journey without opening the app.

#### Acceptance Criteria

1. THE Home_Screen_Widget on iOS SHALL be implemented using SwiftUI and WidgetKit.
2. THE Home_Screen_Widget on Android SHALL be implemented as a native Android widget.
3. THE Home_Screen_Widget SHALL read the relationship start date from local device storage (AsyncStorage or equivalent).
4. THE Home_Screen_Widget SHALL display a running counter in the format "X Days Together" where X is the number of days since the relationship start date.
5. WHEN the app is opened, THE app SHALL update the local storage with the latest relationship start date from Supabase_DB to keep the widget accurate.
6. WHERE the user is a Premium_User, THE Home_Screen_Widget SHALL offer premium widget themes (additional color schemes and layouts).
7. WHILE the user is a Free_User, THE Home_Screen_Widget SHALL display the default theme only.
8. THE Settings_Screen SHALL include instructions for installing the Home_Screen_Widget on both iOS and Android.

### Requirement 15: Deep Connection Conversation Deck (Premium)

**User Story:** As a couple, we want a deck of intimate conversation starters, so that we can deepen our connection through meaningful questions.

#### Acceptance Criteria

1. WHEN the user navigates to Tab 4 (Connection), THE Conversation_Deck SHALL load conversation prompts from a hardcoded local JSON file at `src/assets/deep_questions.json`.
2. THE Conversation_Deck JSON file SHALL contain a minimum of 300 conversation starter prompts.
3. WHEN the Conversation_Deck loads, THE Conversation_Deck SHALL shuffle all prompts using the Fisher-Yates algorithm before displaying them.
4. THE Conversation_Deck SHALL present prompts as swipeable or tap-to-flip cards built with react-native-reanimated.
5. WHERE the user is a Premium_User, THE Conversation_Deck SHALL allow full access to swipe through all cards.
6. WHILE the user is a Free_User, THE Conversation_Deck SHALL display a single preview card followed by a locked card with a "Lock" icon.
7. WHEN a Free_User taps the locked card, THE Conversation_Deck SHALL display the Paywall_Modal.
8. THE Conversation_Deck SHALL not fetch prompts from Supabase_DB or any remote source; all data SHALL reside in the local JSON file.

### Requirement 16: Unlimited Everything and Custom Stickers (Premium)

**User Story:** As a premium user, I want all item limits removed and the ability to use my own photos as calendar stickers, so that I get the full WeDo experience.

#### Acceptance Criteria

1. WHERE the user is a Premium_User, THE Bucket_List SHALL allow unlimited items with no cap.
2. WHERE the user is a Premium_User, THE Sticker_Drawer SHALL display a "Custom Photo" tab in addition to the default stickers.
3. WHEN a Premium_User selects the "Custom Photo" tab, THE Sticker_Drawer SHALL allow the user to pick a photo from the device gallery and crop it into a square Custom_Sticker.
4. WHEN a Custom_Sticker is created, THE Supabase_Storage SHALL store the cropped image at `{relationship_id}/stickers/{sticker_id}.jpg`.
5. THE Custom_Sticker SHALL be draggable onto the Sticker_Calendar using the same drag-and-drop mechanic as Default_Sticker images.
6. WHILE the user is a Free_User, THE Sticker_Drawer SHALL hide the "Custom Photo" tab.
7. WHEN a Free_User taps the area where the "Custom Photo" tab would appear, THE Sticker_Drawer SHALL display the Paywall_Modal.
8. WHERE the user is a Premium_User, THE Home_Screen_Widget SHALL unlock premium widget themes.

### Requirement 17: Entitlement Logic and RevenueCat Integration

**User Story:** As a user, I want a simple one-time purchase to unlock all premium features, so that I can enjoy the full app without recurring charges.

#### Acceptance Criteria

1. THE RevenueCat SDK SHALL manage a single $4.99 non-consumable in-app purchase mapped to the entitlement `premium_lifetime`.
2. WHEN the app loads, THE app SHALL call RevenueCat getCustomerInfo() and store the resulting premium status as `isPremium: true` or `isPremium: false` in the Zustand_Store.
3. WHEN a Free_User attempts to add an 11th Bucket_List item, THE Bucket_List SHALL display the Paywall_Modal.
4. WHEN a Free_User taps the microphone icon on a Memory_Entry, THE Adventure_Log SHALL display the Paywall_Modal.
5. WHEN a Free_User taps the locked Conversation_Deck card, THE Conversation_Deck SHALL display the Paywall_Modal.
6. WHEN a Free_User taps the "Custom Photo" tab in the Sticker_Drawer, THE Sticker_Drawer SHALL display the Paywall_Modal.
7. WHEN the user completes the purchase in the Paywall_Modal, THE RevenueCat SDK SHALL verify the transaction and THE Zustand_Store SHALL update `isPremium` to true.
8. WHEN `isPremium` changes to true, THE app SHALL immediately unlock all premium features without requiring a restart.
9. THE Settings_Screen SHALL provide a "Restore Purchases" button that calls RevenueCat restorePurchases() and updates the Zustand_Store accordingly.
10. IF the RevenueCat purchase verification fails, THEN THE Paywall_Modal SHALL display the message "Purchase could not be verified — please try again."
11. IF the RevenueCat restore call returns no active entitlements, THEN THE Settings_Screen SHALL display the message "No previous purchase found."

### Requirement 18: Navigation Architecture

**User Story:** As a user, I want intuitive tab-based navigation with smooth transitions, so that I can move between features quickly and naturally.

#### Acceptance Criteria

1. THE Root_Navigator SHALL be implemented as a react-navigation Native Stack containing the OnboardingStack, MainTabNavigator, Paywall_Modal, and AddToListModal.
2. THE MainTabNavigator SHALL be implemented as a react-navigation Bottom Tab Navigator with five tabs: Timeline (Tab 1), Calendar (Tab 2), Bucket List (Tab 3), Connection (Tab 4), and Settings (Tab 5).
3. THE MainTabNavigator SHALL display a persistent bottom tab bar with outlined icons.
4. WHEN a tab is active, THE MainTabNavigator SHALL highlight the active tab icon and label in Soft Coral (#FF7F50).
5. WHEN a tab is inactive, THE MainTabNavigator SHALL display the tab icon and label in a muted gray tone.
6. THE Paywall_Modal and AddToListModal SHALL use full-screen slide-up modal presentation style for high-focus tasks.
7. WHEN the user completes authentication and pairing, THE Root_Navigator SHALL replace the OnboardingStack with the MainTabNavigator (the user cannot navigate back to onboarding).
8. THE Settings_Screen (Tab 5) SHALL provide access to: user profile, Partner info, app preferences, subscription management, Privacy_Vault toggle, theme selection, and Home_Screen_Widget installation instructions.
9. WHEN the user taps the "Indecision Wheel" icon on the Bucket List tab, THE Root_Navigator SHALL push the WheelScreen as a stack screen on top of the MainTabNavigator.
10. THE MainTabNavigator SHALL default to the Timeline tab (Tab 1) on initial load after pairing.
